# Database Architecture

## 1. ERD explanation

The HRMIS database uses a tenant-first relational model. `tenants` is the root entity for client companies. Most operational entities carry `tenant_id` and are queried only through tenant-scoped helpers unless the user is `SUPER_ADMIN`.

Core relationships:

- `tenants 1..n users`
- `tenants 1..n departments`
- `tenants 1..n employees`
- `users n..m roles` through `user_roles`
- `roles n..m permissions` through `role_permissions`
- `employees n..1 departments`
- `employees n..1 employees` for reporting manager hierarchy
- `employees 1..n emergency_contacts`
- `employees 1..n job_histories`
- `employees 1..n documents`
- `employees 1..n leave_balances`
- `employees 1..n leave_requests`
- `employees 1..n attendance_records`
- `onboarding_templates 1..n onboarding_tasks`
- `employees n..m onboarding_tasks` through `employee_onboarding_tasks`
- `users 1..n notifications`
- `audit_logs` references tenant, actor, entity type, and entity id for immutable traceability.

## 2. PostgreSQL table design

All primary keys use UUID values generated by PostgreSQL `gen_random_uuid()` in migrations and by Python `uuid4()` in SQLAlchemy models. The initial migration creates `pgcrypto` to support UUID generation.

### Platform/global tables

| Table | Purpose | Notes |
|---|---|---|
| `permissions` | Canonical permission catalog | Unique `code` |
| `roles` | System role catalog | Unique `name`, `is_system` |
| `role_permissions` | Role-permission join | Unique `(role_id, permission_id)` |

### Tenant and user access tables

| Table | Purpose | Tenant strategy |
|---|---|---|
| `tenants` | Client companies | Root table |
| `users` | Login identities | Nullable `tenant_id` only for platform super admins |
| `user_roles` | User-role assignments | Carries `tenant_id` for auditability |
| `notifications` | User notifications | Required `tenant_id` |

### HR operations tables

| Table | Purpose | Key constraints |
|---|---|---|
| `departments` | Tenant org structure | Unique `(tenant_id, name)` and `(tenant_id, code)` |
| `employees` | Employee system of record | Unique `(tenant_id, employee_number)` and `(tenant_id, email)` |
| `emergency_contacts` | Emergency contacts | FK to employee and tenant |
| `job_histories` | Employment history | Date range check |
| `documents` | Secure document repository | Signature/access/status checks, checksum, expiry tracking |
| `leave_types` | Tenant leave policy types | Unique `(tenant_id, name)` |
| `leave_balances` | Accrual and balance records | Unique `(tenant_id, employee_id, leave_type_id, year)` |
| `leave_requests` | Leave workflow | Status/date/positive-days checks |
| `attendance_records` | Attendance logs | Unique `(tenant_id, employee_id, work_date)` |
| `onboarding_templates` | New-hire templates | Unique `(tenant_id, name)` |
| `onboarding_tasks` | Template tasks | Assignee role checks |
| `employee_onboarding_tasks` | Assigned tasks | Unique `(tenant_id, employee_id, task_id)` |
| `audit_logs` | Audit trail | Tenant, actor, entity, action, JSON metadata |

## 3. SQLAlchemy models

The models are defined under `backend/app/models/`. They use:

- `GUID` TypeDecorator for native PostgreSQL UUIDs and SQLite-compatible tests.
- `TimestampMixin` for `created_at` and `updated_at`.
- `SoftDeleteMixin` for soft-deletable entities.
- `TenantMixin` for tenant-scoped tables.
- Explicit relationships for employees, documents, leave, onboarding, users, roles, permissions, audit logs, and notifications.

## 4. Alembic migration scripts

- `001_initial_schema.py`: Creates all tables, UUIDs, FKs, check constraints, timestamps, soft-delete columns, and `pgcrypto`.
- `002_add_indexes_and_constraints.py`: Adds operational and analytics indexes, including employee full-text search.
- `003_seed_default_roles.py`: Seeds system roles and permissions with idempotent `ON CONFLICT` logic.

## 5. Seed data for roles and permissions

Seeded roles:

- `SUPER_ADMIN`
- `HR_CONSULTANT`
- `CLIENT_ADMIN`
- `MANAGER`
- `EMPLOYEE`

Seeded permissions include tenant, user, employee, document, leave, attendance, onboarding, and dashboard capabilities. The same permissions exist in `app/services/rbac_service.py` and in migration `003_seed_default_roles.py`.

## 6. Indexing strategy

Indexes are optimized for tenant-scoped list pages, approval queues, dashboards, document compliance alerts, and analytics-ready aggregations:

- `users(tenant_id, is_active)`
- `employees(tenant_id, deleted_at, employment_status)`
- `employees(tenant_id, department_id)`
- `employees(tenant_id, manager_id)`
- GIN full-text employee search over name, email, and employee number
- `documents(tenant_id, document_type, status)`
- `documents(tenant_id, expiry_date)`
- `leave_requests(tenant_id, status, start_date, end_date)`
- `attendance_records(tenant_id, work_date)`
- `employee_onboarding_tasks(tenant_id, status, due_date)`
- `audit_logs(tenant_id, entity_type, created_at)`
- `notifications(user_id, read_at)`

## 7. Data retention and soft-delete strategy

Soft deletes are used for tenants, users, departments, employees, and documents. These records are excluded by default at query level but preserved for audit, compliance, and recovery.

Recommended retention rules:

- Audit logs: retain at least 7 years or per client contract/regulation.
- Employee records: retain according to applicable employment law and client policy.
- Documents: retain by document type and expiry/termination date.
- Attendance and leave records: retain at least statutory minimums; longer for analytics if anonymized.
- Hard deletion: only through an explicit admin/legal workflow with audit log entry.

## 8. Tenant isolation rules

Application rules:

1. All tenant-specific tables include `tenant_id`.
2. All list/detail/update/delete endpoints use `tenant_query(model)` unless they are platform-global catalog routes.
3. Non-super-admin users cannot override `tenant_id` in API payloads.
4. Foreign-key references are validated against the current tenant before inserts/updates.
5. Document access is checked by both tenant and role/access level.
6. Frontend role-aware navigation is UX only; backend decorators enforce permissions.

Future hardening:

- Add PostgreSQL Row Level Security policies for defense-in-depth.
- Add tenant-aware object storage buckets or prefixes.
- Add per-tenant encryption keys for sensitive document classes.
